# Docnav

Docnav 是 CLI-first 的结构化文档导航系统。它给大模型提供一种更接近“翻文档”的路径：先拿到文档地图，再按稳定位置读取证据，证据不够时沿结构继续扩大范围。

核心路径只有一条：

```text
outline -> ref -> read
```

`outline` 返回章节结构和可继续定位的 `ref`；`read` 只读取当前判断需要的区域。调用方不需要理解 `ref` 的内部格式，只需要把它原样交还给 adapter。这样，目录关系、读取位置和下一步扩展路径会留在流程里，正文不会在一开始全部进入上下文。

## 为什么做 Docnav

先把问题放到 agent 的真实工作流里看。Docnav 要解决的不是“文档能不能被读到”，而是模型在有限上下文里怎样探索文档，才能把更多窗口留给后续真正会用到的规则、事实和证据。

### 窗口先被工作环境占用

模型真正开始完成任务前，通常已经加载了大量基础上下文：系统提示词、工具调用说明、skill、插件、项目的 `AGENTS.md`、用户当前需求，以及为了理解项目而产生的探索记录。真正开始实现、审查或决策之前，窗口里已经有很多必须存在但不直接解决问题的内容。

在这样的前提下，Docnav 关注的是上下文利用率。探索阶段少把无效正文写入窗口，一个完整上下文窗口里就能容纳更多后续会被使用的信息。

### 文档探索比代码探索更粗

基础上下文之外，最容易继续膨胀的是项目探索。代码生态已经有很多结构化探索工具，可以按符号、调用链、文件树或搜索结果逐步缩小范围；文档却经常只剩“全文读取”和“猜测读哪段”两种方式。

在 AI 主导的项目里，文档先行、规范多、ADR 多、设计说明多是常态。把所有文档完整读入，会让上下文窗口更快接近上限，也会降低有效信息在窗口里的占比。很多文档只是为了判断“是否相关”而被读取，最后并不会真正影响实现、审查或决策。

### 先看结构，再读证据

Docnav 的做法是把文档探索从“先读完再判断”改成“先看结构，再按需取证”。`outline`、目录关系和 `ref` 承担文档地图的角色，正文 token 只在当前判断需要证据时进入上下文。

这个过程不会让系统提前准确知道什么内容相关。它只是把不确定性拆成可继续的步骤：先用目录、层级和轻量 metadata 形成假设，再通过 `ref` 读取局部内容；如果证据不足，就沿父子章节、相邻章节、搜索结果或批量读取继续扩大范围。

### 不拆碎人类文档

如果只服务 agent，手动拆分文档也能模拟一部分结构化阅读。但项目往往同时服务人类和 AI。人类需要统一的文档入口、连续的叙述和完整的上下文；文档拆得过碎，会降低人的阅读和维护体验。

Docnav 试图在两者之间给出一个小工具：文档仍然可以按人类友好的方式组织，agent 则可以按结构化路径渐进阅读。

## 成本分两段看

上面的目标是提高上下文利用率，不是承诺当前读取调用一定更便宜。结构化阅读会少读一部分正文，但也会引入 `outline`、`read`、工具结果包装和多轮判断；已有上下文越长，多出来的判断轮次越明显。所以成本要分两段看：先看当前探索调用，再看这些上下文在后续执行阶段怎样继续占用窗口。

### 当前调用通常不占便宜

先把当前探索调用的比较口径收窄。这里只比较输入 token，不比较输出 token，并按普通输入和缓存输入价格 `10:1` 粗估。输出 token 会受到模型解释长短、工具结果转述方式和任务类型影响，放在一起比较容易把问题搅乱。

要写成公式，先把两条路径里的输入来源拆开。`结构化新增输入` 不只是正文片段，还包括目录、工具包装、`ref`、结果 metadata 和实际读到的内容；`判断轮数` 指结构化路径里模型读取工具结果后继续决策的次数。

$$
\text{直接全文读取} \approx \frac{\text{已有上下文}}{10} + \text{全文正文}
$$

$$
\text{结构化读取} \approx \frac{\text{判断轮数} \times \text{已有上下文}}{10} + \text{结构化新增输入}
$$

这两个式子说明，结构化阅读要在当前读取调用里更便宜，省下的正文必须先覆盖多出来的判断轮次：

$$
\text{全文正文} - \text{结构化新增输入} > \frac{(\text{判断轮数} - 1) \times \text{已有上下文}}{10}
$$

而结构化路径本身还要承担这些新增输入：

$$
\text{结构化新增输入} \approx \text{工具包装} + \text{目录} + \text{ref 和 metadata} + \text{实际读入正文}
$$

把这个门槛放回真实文档里时，还需要先把“全文正文”估成 token 量级。这里的文件大小只适合做粗略参照，不能当成稳定换算公式。中文说明型 Markdown 的 `1 KB` 常见落点是几百 token，正文越偏中文、标点和 Markdown 结构越多，越可能靠近高位；英文、代码和重复结构多时会变化，最终仍以实际 tokenizer 为准。拿本仓库的 [文档导航](docs/navigation.md) 做参照：它约 `7 KB`、90 行，可以粗估为 `2K` 到 `4K` token。一个类似密度的 120 行、`13 KB` 中文 Markdown，大致可以先估成 `4K` 到 `8K` token。

有了这个量级，短文档的劣势会很明显：哪怕只读了少量正文，目录、工具包装、`ref`、metadata 和至少一次局部读取加起来，也可能接近全文的三四成。这个比例不是固定工具成本，而是结构化路径本身会产生额外输入。

### 60K：短文档

先把已有上下文放在 `60K`。这在这种工作流里只能算较短上下文。前面那个 120 行、`13 KB` 的中文 Markdown，如果全文大致是 `4K` 到 `8K` token，可以先取中间量级 `6K` 估算；结构化新增输入按全文的 `40%` 粗估，就是约 `2.4K` token。假设结构化路径需要两轮判断：

$$
\text{直接全文读取} \approx 60K / 10 + 6K = 12K
$$

$$
\text{结构化读取} \approx 2 \times 60K / 10 + 2.4K = 14.4K
$$

这个例子说明，短文档能省下的正文不多，多出来的一轮缓存上下文已经足够吃掉结构化阅读省下的部分。只看这次读取，直接全文读取的输入成本更低。

### 120K：项目文档

再把已有上下文拉到 `120K`，并换成更大的项目文档。项目里的 [core-cli spec](openspec/specs/core-cli/spec.md) 约 358 行、20.8 KB，目录展开后是十几个 H3 级需求。类似密度的中文 Markdown 可以粗估为 `7K` 到 `12K` token；这里取 `10K` 做中间量级。

直觉上，文档变大后结构化阅读看起来更容易占优；但已有上下文也变长了，如果目录层级不够细，结构化路径还会读入不少正文。先看直接全文读取：

$$
\text{直接全文读取} \approx 120K / 10 + 10K = 22K
$$

再看结构化路径。假设它需要三轮判断。目录和工具包装按约 `2K` 粗估；由于目录层级只有三层、每个需求块仍然较长，最后还要读入约 `20%` 到 `40%` 的正文，也就是 `2K` 到 `4K`。结构化新增输入约为 `4K` 到 `6K`：

$$
\text{结构化读取} \approx 3 \times 120K / 10 + (4K \text{ 到 } 6K) = 40K \text{ 到 } 42K
$$

这个例子说明，长上下文会把判断轮次的成本放大。只看这次读取，直接全文读取仍然更便宜；不是因为结构化阅读没有少读正文，而是少读的正文没有覆盖三轮缓存上下文和结构化新增输入。

### 后续调用会继续付费

前面两个例子都显示，当前读取阶段并不是结构化阅读最容易占优的地方。它真正想换的是后续执行阶段的上下文质量。

直接全文读取可以用更低的当前输入成本完成探索，但未使用的正文会进入历史上下文。后续每一次调用工具、读文件、跑测试或继续判断，这些正文都会继续占用窗口；即使命中缓存，也仍然会按约 `1/10` 的价格参与输入计费。随着后续调用次数增加，这部分成本会逐轮累积。

如果结构化阅读让一部分无效正文没有常驻在上下文里，后续每轮就可能少付一段缓存输入成本，也少占同样数量的上下文窗口：

$$
\text{后续累计少付输入成本} \approx \text{后续轮数} \times \frac{\text{少常驻的无效正文}}{10}
$$

这个收益无法承诺，也很难严格证明，因为它取决于后续任务会跑多少轮、缓存是否命中、被省掉的正文原本会不会继续留在窗口里。保守地说，这只能算一点心理安慰：当前读取调用可能更贵，但换来的历史上下文更接近“目录、证据位置和相关片段”，而不是整段已经排除掉的正文。

### 什么时候值得用

所以选择读取方式时，不能只看文档大小。还要看已有上下文长度、后续还要围绕它工作多少轮、相关内容是否集中、目录能不能把实际读取比例压下来。

结构化阅读更适合长文档、相关内容稀疏、需要多次回溯、后续还会继续执行多轮任务的场景。尤其在长上下文下，全文读取虽然可能在当前读取阶段更便宜，但无关正文会继续占窗口、继续产生缓存输入成本，并随着后续调用次数累积。

直接全文读取更适合短文档、一次性问题、上下文还不长、或者全文本来就需要整体理解的场景。此时结构化阅读省不下多少正文，额外判断轮次和工具包装会放大当前输入成本。

这个判断也把体验优化的方向带出来了：结构化阅读要更可用，不能只靠“先看目录”这个想法，还要控制额外输入，减少不必要的判断轮次，并让读错、补读和扩大范围的过程更顺。

## 体验优化

Docnav 会在高频阅读路径上主动做一些小组合：小文档直接返回全文，多个 `ref` 批量读取，搜索命中时顺手展开，必要时补一点父章节或相邻章节。它们会多带一点受控的输入或工具行为，但能换掉更贵的串行判断、手动拼接和重复读取。结构化阅读如果每一步都要求 agent 自己判断、自己保存 `ref`、自己再发起 `read`，理论上的上下文收益很容易被实际使用成本抵消。

小文档直返是最典型的例子。小文档本来就省不下太多正文，如果还先返回目录，再让 agent 判断一次、调用一次 `read`，多出来的往返和判断成本会很明显。直接返回全文会多给一点普通输入，但这个代价是可控的；换来的结果是少一轮工具调用、少一次判断延迟，也少一次因为目录粒度不足而读错位置的机会。

批量读取、搜索命中展开、父章节和相邻章节补读也是同一类优化。它们不一定让单次工具结果最小，但能减少 agent 在多个 `ref` 之间来回调度的成本。很多时候，多带一点明确相关的上下文，比让模型再走一轮“读取、判断、再读取”更划算。

这些体验优化主要放在 core 或 SDK 层完成。`outline`、`read`、`find` 和 `info` 仍然是基础能力，core/SDK 负责把常见路径包顺；adapter 只处理自己格式的识别、解析、`ref` 和单次读取语义。这样做的好处是，常见路径可以继续打磨，格式接入也不会被 agent 的具体阅读策略绑死。

## 扩展边界

体验层和格式层分开以后，扩展其它信息源会更自然。Docnav 不要求每种格式都理解 agent 的完整策略，只要求它能提供结构、稳定位置和局部读取语义。Markdown 只是第一个接入的格式；其它来源只要能被整理成“地图、稳定位置、局部读取、继续导航”这套模型，就可以复用同一条阅读路径。

按这个扩展模型，未来可以覆盖的内容包括：

- JSON、YAML、TOML 这类配置和数据文件，可以按 path、key、数组范围或 schema 片段阅读。
- 代码可以按文件、符号、注释块、测试或语义区域阅读。
- Excel 和表格可以按 sheet、表格区域、行列范围、命名区域或摘要阅读。
- 数据库可以按 schema、table、column、index、样例行或查询计划阅读。
- HTML、PDF、日志、API 描述、导出的配置和运行报告，也可以在有稳定定位方式时纳入同一类探索。

这些来源表面差异很大，但接入点相同：把原始内容转换成可导航的结构、稳定引用和按预算读取的片段。Docnav 从 Markdown 起步，保留的是继续接入其它信息源的工程边界。

## 获取 Beta

首个公开 Beta 版本为 `v0.1.0-beta.1`，支持 Linux `x86_64-unknown-linux-gnu` 和 Windows `x86_64-pc-windows-msvc`。公开 prerelease 创建后，可从 [GitHub prerelease](https://github.com/zxyycom/docnav/releases/tag/v0.1.0-beta.1) 下载与平台匹配的 binary 和同名 checksum：

| 平台 | Binary | Checksum |
| --- | --- | --- |
| Linux | `docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu` | `docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu.sha256` |
| Windows | `docnav-v0.1.0-beta.1-x86_64-pc-windows-msvc.exe` | `docnav-v0.1.0-beta.1-x86_64-pc-windows-msvc.exe.sha256` |

在下载目录校验并启动 Linux binary：

```bash
sha256sum --check docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu.sha256
chmod +x docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu
./docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu version
./docnav-v0.1.0-beta.1-x86_64-unknown-linux-gnu outline "/path/to/document.md"
```

在 PowerShell 中校验并启动 Windows binary：

```powershell
$asset = "docnav-v0.1.0-beta.1-x86_64-pc-windows-msvc.exe"
$expected = (((Get-Content "$asset.sha256" -Raw).Trim() -split "\s+")[0]).ToLowerInvariant()
$actual = (Get-FileHash $asset -Algorithm SHA256).Hash.ToLowerInvariant()
if ($actual -ne $expected) { throw "SHA-256 mismatch" }
& ".\$asset" version
& ".\$asset" outline "C:\path\to\document.md"
```

该 prerelease 将只交付上述两个 x86_64 target 的裸 binary 和对应 SHA-256；不提供归档包、installer、binary signature 或平台公证。完整命令面见 [CLI](docs/cli.md)，制品与校验边界见 [发布包验证](docs/testing/release.md)。
