# Docnav 编码规范

本文档定义 `Docnav` 实现代码的长期编码原则。它约束工程判断、边界处理、职责分层和验收标准，不替代产品规范，也不预设具体模块名、函数名或重构方案。

具体命令、协议字段、adapter 契约、ref 语义、schema 形状和测试矩阵，以 [文档导航](navigation.md) 的“如何阅读这些文档”指向的主规范为准。编码规范只回答一个问题：实现这些契约时，代码应该保持什么性质。

读者是正在修改代码的实现者、AI coding agent 和 reviewer。使用时先按任务读取主规范，再用本文件约束实现取舍；它用于改动前的实现归属判断、实现中的边界处理，以及交付前的自检。

---

## 1. 文档边界与使用方式

编码规范只负责实现代码的通用质量规则：如何识别规范 owner 与实现归属、如何处理边界、如何用类型表达领域含义、如何组织文件和模块、如何选择验证层级，以及交付前检查什么。

编码规范不定义产品职责、协议字段、adapter 契约、schema 形状、错误语义、输出格式或测试矩阵。这些规则由 [文档导航](navigation.md) 指向的主规范定义；本文件只引用它们，不复制完整规则。

实现前先确认三件事：

1. 本次改动的规则来源是哪份主规范，并落到哪个制品、语义层和实现归属。
2. 本次改动跨过哪些边界，例如 CLI、配置、文件系统、进程、JSON、schema、adapter 输出或测试验证脚本。
3. 本次交付需要哪些证据证明受影响边界仍成立。

验收标准：reviewer 能从改动说明或代码位置看出本次改动的规范 owner、实现归属、影响边界和验证方式；不能说明规则来源或代码归属时，应先澄清边界再实现。

---

## 2. 规范 owner 与实现归属先行

实现代码必须服从主规范定义的职责边界和稳定契约。规范 owner 是规则来源文档；实现归属是承载代码的制品、模块、文件或脚本。两者需要对齐，但不是同一个东西。

新增逻辑先进入最明确的现有实现归属；只有职责、变化原因、复用边界或命名空间清晰时，才新增文件、模块、crate 或共享层。

跨实现归属协作必须通过主规范定义的边界、显式类型或稳定输出完成。接入层只做本层负责的解析、校验、归一化和映射；属于下游组件的 opaque 值或业务语义原样传递，不重新生成、解析或判断。

共享代码不能用来绕过实现归属。只有多个实现归属共享同一不变量、错误映射或数据结构，且该共享职责有清晰命名和验证入口时，才抽出公共模块。

验收标准：新增或修改的代码能回答“哪份规范定义这条规则、哪段代码负责实现、谁消费这个结果、谁负责失败反馈”。

---

## 3. 边界代码显式

用户输入、CLI 参数、配置、路径、文件系统、进程、JSON、schema 和 adapter 输出都属于边界。边界代码负责解析、校验、归一化和错误映射；内部逻辑只处理已经归一化的领域对象。

边界失败必须给调用方明确反馈：

1. 产品入口返回稳定错误、明确退出码或可行动诊断。
2. 验证脚本失败时说明文件、字段和原因。
3. 机器输出和诊断输出保持各自通道职责。

边界规则可以保留否定式，但必须配套正向做法：

边界：不通过空结果、默认 adapter、默认 ref、猜测 content type 或静默跳过掩盖失败。

做法：把失败转换为对应稳定错误、验证失败、候选证据或诊断信息，并保留足够的定位依据。实现可恢复兜底时，必须能追溯到主规范定义的输出层规则，不能在业务逻辑中静默丢弃失败。

验收标准：任何失败路径都能回答“调用方下一步该看什么、改什么或重试什么”。

## 4. 类型表达领域含义

跨模块传递领域含义时，代码应使用能表达业务含义和不变量的类型。裸字符串、通用 JSON 值和元组只适合协议镜像、opaque options、错误 details、脚本校验和局部 glue code。

类型设计遵循三条原则：

1. 协议镜像类型保持稳定序列化形状，不把格式私有语义塞进共享协议。
2. 业务逻辑入口接收已经校验过的领域值，不重复解析原始输入。
3. 公开 API 通过返回值表达普通失败，不因调用方可传入的普通非法值 panic。

稳定协议字段、operation、capability、错误码、退出码、schema 名、示例路径、固定诊断前缀和其它跨函数复用的字符串或数字，应使用枚举、新类型、常量模块或职责内配置中心表达。配置中心按实现归属拆分，不建立跨制品的全局杂物常量池。

测试 fixture、一次性局部标签和只在单个小作用域中用于构造示例的数据可以保留直写，但不能把稳定规则、跨边界字段或错误映射散落为魔法字符串。

脚本代码同样适用本节。`scripts/` 和 `test/` 下跨模块传递的任务配置、schema 结果、进程执行结果、质量观测结果和 smoke state 应有显式 TypeScript 类型；脚本运行方式、文件扩展名和类型检查验证入口由 [工程工具链](tooling.md) 拥有。

验收标准：阅读函数签名时，可以区分“原始输入”“协议字段”“业务对象”和“格式私有参数”。

## 5. 文件与模块按变化原因组织

文件、模块、函数和脚本应围绕单一变化原因组织。编码规范不规定固定目录方案；具体拆分应服从当前代码形状和相邻模块惯例。

新增代码先进入最明确的现有实现归属。文件大小、函数长度和质量检查结果只是观察信号，不是拆分依据；拆分要服务于职责、变化原因、复用边界或局部可读性。

只服务一个父模块的 helper、wrapper、常量和小类型，优先作为父模块内部实现；多个同级文件若只能靠共同前缀表达分组，优先把该分组表达为目录或模块，除非项目惯例、外部契约、生成规则或测试编号要求保持当前形态。

Rust 模块拆分时，优先采用“同名入口文件 + 同名目录”的形态，例如由 `reader.rs` 声明并组织 `reader/` 下的子模块。新增模块入口使用同名文件，不使用 `mod.rs`；`lib.rs`、`main.rs` 等 crate root 按 Rust 约定保留。

需要拆分的信号：

1. 同一文件同时承载多个制品职责、多个语义层或多个边界类型。
2. 新增功能需要反复修改无关代码。
3. 测试、fixture、解析、输出和错误映射互相遮蔽，难以局部阅读。
4. 代码评审无法用一个简短句子说明该单元的主职责。

验收标准：文件路径和文件名能表达实现归属；新增或修改的代码职责清晰，局部改动不会迫使读者理解无关制品或无关语义层。

## 6. 稳定规则只引用规范 owner

代码和验证脚本用于实现或校验规则，不成为新的规则来源。示例用于证明语义和字段形状，不替代主规范或 schema。稳定字段、稳定错误、输出语义、adapter 行为、schema 形状和测试矩阵必须能追溯到对应规范 owner。

当实现发现规则缺失、冲突或过期时，先判断规范 owner：

1. 产品职责、协议语义、输出语义和 adapter 契约由主规范定义。
2. JSON 字段形状由 schema 定义；示例语义校验由示例承载。
3. 测试层级、覆盖目标和 case 归属由测试规范和 case 账本定义。
4. 脚本语言、包管理和验证入口由工具链文档定义。

涉及 contract surface 时，按影响面同步主规范、schema、示例、代码和验证材料；未触及的材料不为“保持一致”而复制无关规则。

验收标准：任何稳定规则变化，都能指出唯一规范 owner，并说明本次改动同步了哪些受影响材料。

---

## 7. 验证按风险选择层级

验证范围随风险和影响面扩展。窄改动用局部测试验证不变量；跨边界改动覆盖调用链、输出层和错误映射；契约改动覆盖对应主规范、schema、示例和语义校验。

触及对应范围时，最低要求：

1. 内部语义有单元测试或等价局部验证。
2. 入口行为有集成测试或等价命令验证。
3. schema、示例和 machine output 保持可编译、可解析、可互相映射。
4. 输出层边界和 stderr/stdout 边界有测试或脚本检查。
5. 纯文档改动至少用 `dnm outline`、局部 diff 或等价方式确认结构和范围。
6. 无法运行的验证必须在交付说明中明确原因和风险。

验收标准：验证能证明本次改动影响的边界仍成立，而不是只证明代码可以编译。

---

## 8. 变更前后自检

实现前确认规范 owner、实现归属、受影响边界和验证方式；提交或交付前检查：

1. 本次改动是否已按 [文档导航](navigation.md) 找到规则的规范 owner，并落到明确实现归属。
2. 职责边界是否仍符合主规范，未在编码规范或实现代码中重新定义 contract。
3. 边界失败是否显式反馈，没有静默兜底。
4. 公开 API 是否通过类型或返回值表达不变量和失败。
5. 跨边界数据的责任方是否清晰，边界层未复制下游组件的解析、生成或语义判断。
6. 涉及文件布局变化时，路径、文件名和分组是否能表达实现归属和独立职责；Rust 新模块入口是否采用同名文件形态。
7. 稳定规则是否只有一个规范 owner；涉及 contract surface 时，是否按影响面同步规范、schema、示例、代码和验证。
8. 验证命令是否按改动范围运行并记录结果。
